## Part 3: llama.cpp evaluation

### 3.1 The premise problem

llama.cpp is a high-performance C++ inference runtime. You asked: "Could modelito integrate llama.cpp as a provider?"

**Answer:** Not cleanly. Here's why.

### 3.2 Architecture mismatch

**modelito's design:**
- Provider Protocol = (list_models, summarize, stream, embed)
- Pure-Python core, optional SDKs (openai, anthropic, google-generativeai, etc.)
- Deterministic shims so tests don't require network or binaries
- Each provider is stateless adapters over an external service or lightweight Python wrapper

**llama.cpp:**
- C++ inference engine with Python bindings (llama-cpp-python)
- Requires precompiled binary or local compilation (not pip install under the hood — it's CFFI with a build step)
- Inference-only; no abstractions for list/enumerate models
- Requires model files on disk (GGUF format)
- Model lifecycle (download, cache, load) is the user's responsibility

### 3.3 Technical barriers

| Barrier | Impact |
|---------|--------|
| **Binary dependency** | Can't be pure-Python optional. llama-cpp-python needs C++ compiler or prebuilt wheels. Breaks the "deterministic fallback" model. |
| **Model file management** | modelito doesn't have a download/cache system. Users must manage GGUF files separately. OllamaProvider delegates this to Ollama itself; llama.cpp would need a parallel system. |
| **No hosted equivalent** | Every other provider (OpenAI, Claude, Gemini, Ollama) has a network service backend. llama.cpp is local-only. This means no easy way to support both local and cloud equivalents in the same codebase structure. |
| **list_models() and streaming** | llama.cpp has no concept of "list available models." You'd need custom logic to scan a directory or registry. Streaming is supported but doesn't map cleanly to the Provider protocol's stream() signature. |
| **Deterministic testing** | Can't stub a llama.cpp provider without mocking the entire inference stack. The shim would be useless. |

### 3.4 What you could do instead

**Option A: Separate llama.cpp shim** (lowest integration lift)
- Create a standalone modelito_llamacpp package that implements the Provider protocol
- Users opt-in: `pip install modelito modelito_llamacpp`
- Handles model file discovery, GGUF caching, deterministic fallback separately
- Decoupled from modelito's release cycle

**Option B: Local-inference plugin** (more structured)
- Add a LocalInferenceProvider base class (or LocalProvider protocol)
- Define model-discovery and cache-directory semantics once
- Implement llama.cpp as one LocalInferenceProvider; leave room for gpt4all, ollama (already there), etc.
- Harder design work upfront but cleaner long-term

**Option C: Stay focused** (recommended)
- Keep modelito as a provider shim for hosted + lightweight local services (Ollama, mock)
- llama.cpp users use llama-cpp-python directly or pick a dedicated local-inference library
- Not every tool needs to do everything

### 3.5 Conclusion on llama.cpp

llama.cpp is excellent software, but it's a **different category** from what modelito does:

- **modelito**: provider abstraction layer + lightweight adapters to hosted services
- **llama.cpp**: standalone inference runtime with optional Python bindings

Integrating llama.cpp into modelito would require compromising:
- The pure-Python, optional-SDK design
- The deterministic-fallback philosophy
- The provider abstraction (llama.cpp is inference, not a chat provider)

**Recommendation:** If local inference is important for your stack, either (a) create a separate modelito_llamacpp adapter, or (b) use llama-cpp-python directly and keep modelito for the hosted-provider abstraction. Cleaner architecture either way.

## Summary

modelito's architecture is sound and its optional-extras + deterministic-fallback philosophy is well-executed. The package is usable and the release infrastructure is modern. 

**Main gaps are hygiene, not design:**
- Documentation silos (dev-deps, provider lists, version notes)
- Release versioning out of sync
- Config file redundancy

**Most issues are 15-minute fixes.** Addressing them will improve clarity for contributors and new users without requiring architectural changes.

**llama.cpp doesn't belong in-tree**, but a separate local-inference adapter is plausible if the need grows.
